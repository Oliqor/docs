# Permissions

Oliqor uses a **bitfield permission system** inspired by Discord. Every role has a `permissions` field stored as a BigInt string, and individual channels can override permissions per role or per member.

## How Permissions Work

Permissions are represented as a single integer where each bit corresponds to a specific permission. A permission is granted when its bit is set to `1`.

```
permissions = "104324673"  →  binary: 110001101100010...001
                                       ↑ bit 27 = VIEW_GUILD_INSIGHTS
```

To check if a permission is granted:

```typescript
const permissions = BigInt("104324673");
const MANAGE_MESSAGES = 1n << 9n;

if (permissions & MANAGE_MESSAGES) {
	// User has MANAGE_MESSAGES
}
```

To combine permissions:

```typescript
const combined = SEND_MESSAGES | ADD_REACTIONS | READ_MESSAGE_HISTORY;
```

## Permission Reference

| Permission                 | Bit       | Value       | Description                                                                                      |
| -------------------------- | --------- | ----------- | ------------------------------------------------------------------------------------------------ |
| `ADMINISTRATOR`            | `1 << 0`  | `1`         | Bypasses all permission checks. Grants full access to everything.                                |
| `CREATE_CHANNELS`          | `1 << 1`  | `2`         | Create new channels in the guild.                                                                |
| `MANAGE_CHANNELS`          | `1 << 2`  | `4`         | Edit or delete channels, manage permission overwrites.                                           |
| `MANAGE_ROLES`             | `1 << 3`  | `8`         | Create, edit, or delete roles. Can only manage roles below their highest role.                   |
| `KICK_MEMBERS`             | `1 << 4`  | `16`        | Remove members from the guild.                                                                   |
| `BAN_MEMBERS`              | `1 << 5`  | `32`        | Ban and unban members.                                                                           |
| `MANAGE_COMMUNITY`         | `1 << 6`  | `64`        | Edit guild settings, manage auto-moderation rules.                                               |
| `VIEW_CHANNEL`             | `1 << 7`  | `128`       | View a channel and read its message history.                                                     |
| `SEND_MESSAGES`            | `1 << 8`  | `256`       | Send messages in text channels.                                                                  |
| `MANAGE_MESSAGES`          | `1 << 9`  | `512`       | Delete messages from other users, pin/unpin messages.                                            |
| `ADD_REACTIONS`            | `1 << 10` | `1024`      | Add reactions to messages.                                                                       |
| `READ_MESSAGE_HISTORY`     | `1 << 11` | `2048`      | Read message history in a channel. Without this, users only see messages sent after they joined. |
| `MENTION_EVERYONE`         | `1 << 12` | `4096`      | Use @everyone and @here mentions.                                                                |
| `ATTACH_FILES`             | `1 << 13` | `8192`      | Upload files and images in messages.                                                             |
| `EMBED_LINKS`              | `1 << 14` | `16384`     | Links posted will have embedded previews.                                                        |
| `USE_VOICE`                | `1 << 15` | `32768`     | Connect to and speak in voice channels.                                                          |
| `MUTE_MEMBERS`             | `1 << 16` | `65536`     | Server-mute other members in voice channels.                                                     |
| `DEAFEN_MEMBERS`           | `1 << 17` | `131072`    | Server-deafen other members in voice channels.                                                   |
| `MOVE_MEMBERS`             | `1 << 18` | `262144`    | Move members between voice channels or disconnect them.                                          |
| `MANAGE_INVITES`           | `1 << 19` | `524288`    | View and delete invites created by others.                                                       |
| `CREATE_INVITES`           | `1 << 20` | `1048576`   | Create invite links to the guild.                                                                |
| `MANAGE_EMOJIS`            | `1 << 21` | `2097152`   | Create, edit, or delete custom emojis.                                                           |
| `VIEW_AUDIT_LOG`           | `1 << 22` | `4194304`   | View the guild's audit log.                                                                      |
| `USE_COMMANDS`             | `1 << 23` | `8388608`   | Use bot slash commands.                                                                          |
| `USE_THREADS`              | `1 << 24` | `16777216`  | Create and participate in public threads.                                                        |
| `CREATE_PRIVATE_THREADS`   | `1 << 25` | `33554432`  | Create private threads.                                                                          |
| `SEND_MESSAGES_IN_THREADS` | `1 << 26` | `67108864`  | Send messages in threads.                                                                        |
| `VIEW_GUILD_INSIGHTS`      | `1 << 27` | `134217728` | Access the guild analytics dashboard.                                                            |

## Default Member Permissions

When a new role is created or a user joins with the `@everyone` role, they receive these permissions by default:

- `VIEW_CHANNEL`
- `SEND_MESSAGES`
- `ADD_REACTIONS`
- `READ_MESSAGE_HISTORY`
- `ATTACH_FILES`
- `EMBED_LINKS`
- `USE_VOICE`
- `CREATE_INVITES`
- `USE_COMMANDS`
- `USE_THREADS`

## The Administrator Permission

The `ADMINISTRATOR` permission (bit 0) is special — it **bypasses all permission checks**. A user with this permission can do anything in the guild regardless of other permission settings or channel overwrites.

The guild **owner** always has implicit administrator access, even if no role grants it.

## Permission Hierarchy

Permissions are computed in this order:

1. **Base permissions**: Start with the `@everyone` role's permissions.
2. **Role permissions**: OR together permissions from all roles the member has.
3. **Administrator check**: If the result includes `ADMINISTRATOR`, grant all permissions.
4. **Channel overwrites (deny)**: Apply any `deny` overwrites for the `@everyone` role, then for the member's roles.
5. **Channel overwrites (allow)**: Apply any `allow` overwrites for the `@everyone` role, then for the member's roles.
6. **Member-specific overwrites**: Apply any deny/allow overwrites for the specific member.

### Channel Permission Overwrites

Channels can have per-role or per-member permission overwrites. Each overwrite has an `allow` and `deny` bitfield:

```json
{
	"id": "role_moderators",
	"type": 0,
	"allow": "512",
	"deny": "0"
}
```

| `type` | Meaning          |
| ------ | ---------------- |
| `0`    | Role overwrite   |
| `1`    | Member overwrite |

**Example**: To make a channel read-only for `@everyone` but allow moderators to send messages:

```
@everyone overwrite:  deny = SEND_MESSAGES (256)
moderator overwrite:  allow = SEND_MESSAGES (256)
```

## Permissions in API Responses

Permissions are always serialized as **strings** (not numbers) to avoid precision loss with large BigInt values:

```json
{
	"id": "role_abc",
	"name": "Moderator",
	"permissions": "104324673"
}
```

When the API returns a member object with a `permissions` field, it represents the **computed** permissions for that member in the current context (all roles + overwrites resolved).
