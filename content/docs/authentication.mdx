# Authentication

The gateway supports three authentication methods depending on the type of client.

## User Authentication (Bearer JWT)

Most endpoints require a JWT passed as a Bearer token. This is the standard auth method for user-facing clients (web, mobile, desktop).

```
Authorization: Bearer <token>
```

The JWT is validated on every request.

### How it works

1. Register an account with `POST /v1/auth/register` or log in with `POST /v1/auth/login`.
2. The server returns a session token (JWT).
3. Attach the token to every API request in the `Authorization` header.
4. The gateway validates the token and processes the request.

---

## Native Auth Endpoints

### Register

```
POST /v1/auth/register
```

Creates a new account with email and password. Returns a session token.

**Request body**:

| Field      | Type   | Required | Description    |
| ---------- | ------ | -------- | -------------- |
| `username` | string | Yes      | Username       |
| `email`    | string | Yes      | Email address  |
| `password` | string | Yes      | User password  |

### Login

```
POST /v1/auth/login
```

Authenticates with email and password. If MFA is enabled, returns `mfa_required: true` and a short-lived `mfa_token` instead of a session token.

**Request body**:

| Field      | Type   | Required | Description   |
| ---------- | ------ | -------- | ------------- |
| `email`    | string | Yes      | Email address |
| `password` | string | Yes      | Password      |

### Logout

```
POST /v1/auth/logout
```

### Social OAuth Login

```
GET /v1/auth/oauth/{provider}
GET /v1/auth/oauth/{provider}/callback
```

`GET /v1/auth/oauth/{provider}` redirects to the provider's OAuth login page. The `redirect_uri` must match an origin in `OAUTH_ALLOWED_ORIGINS`.

| Parameter      | Type   | Required | Description                        |
| -------------- | ------ | -------- | ---------------------------------- |
| `redirect_uri` | string | No       | URI to redirect to after auth      |

On callback, the user is authenticated and a session is created.

## Multi-Factor Authentication (MFA)

### MFA Status

```
GET /v1/auth/mfa/status
```

Returns whether MFA is enabled for the current user.

### Setup MFA (TOTP)

```
POST /v1/auth/mfa/setup
```

Initiates the TOTP setup process. Returns a QR code / secret for authenticator apps.

### Enable MFA

```
POST /v1/auth/mfa/enable
```

**Request body**:

| Field  | Type   | Required | Description                             |
| ------ | ------ | -------- | --------------------------------------- |
| `code` | string | Yes      | TOTP code from authenticator to confirm |

### Verify MFA

```
POST /v1/auth/mfa/verify
```

Completes login by verifying a TOTP code or backup code. Accepts the `mfa_token` returned by `POST /v1/auth/login`.

**Request body**:

| Field       | Type   | Required | Description                         |
| ----------- | ------ | -------- | ----------------------------------- |
| `mfa_token` | string | Yes      | Temporary MFA token from login step |
| `code`      | string | Yes      | TOTP code from authenticator        |

### Disable MFA

```
POST /v1/auth/mfa/disable
```

**Request body**:

| Field      | Type   | Required | Description           |
| ---------- | ------ | -------- | --------------------- |
| `password` | string | Yes      | Current user password |

### Regenerate Backup Codes

```
POST /v1/auth/mfa/backup-codes/regenerate
```

**Request body**:

| Field  | Type   | Required | Description       |
| ------ | ------ | -------- | ----------------- |
| `code` | string | Yes      | Current TOTP code |

## Bot Authentication (Bot-Token)

Bot applications use a static token for authentication.

```
Bot-Token: <bot_token>
```

### Obtaining a bot token

1. Create an application via `POST /v1/applications`.
2. The response includes a `token` field — **this is shown only once**. Store it securely.
3. If you lose the token, generate a new one with `POST /v1/applications/{botId}/reset-token` (this invalidates the old token immediately).

### Token format

Bot tokens are 64-character lowercase hex strings (32 random bytes, hex-encoded).

### Which endpoints use Bot-Token?

- `PUT /v1/applications/{botId}/commands` — Register global commands
- `PUT /v1/applications/{botId}/guilds/{guildId}/commands` — Guild-specific commands
- `POST /v1/webhooks/{webhookId}/{webhookToken}` — Interaction follow-up messages
- `PATCH /v1/webhooks/{webhookId}/{webhookToken}/messages/@original` — Edit original response
- `DELETE /v1/webhooks/{webhookId}/{webhookToken}/messages/@original` — Delete original response

## OAuth2 Authentication

Third-party applications can authenticate users via the OAuth2 authorization code flow. After authorization, the application receives an access token scoped to specific permissions.

```
Authorization: Bearer <oauth_access_token>
```

OAuth2 access tokens follow the format: `olq_<64-char hex>`

See the [OAuth2 guide](./oauth2) for the full authorization flow, scopes, and token management.

## ConnectRPC Gateway Authentication

The real-time gateway authenticates via the same Bearer JWT, passed in the `token` field of the `IdentifyRequest` message:

```typescript
const stream = client.identify({
	token: "<jwt>",
	intents: GatewayIntents.GUILDS | GatewayIntents.GUILD_MESSAGES,
});
```

Bots can also connect to the gateway by passing a `Bot-Token` header on the HTTP/2 request.

---

## Error Responses

All authentication failures return a standard error:

| Status | Meaning                                                             |
| ------ | ------------------------------------------------------------------- |
| 401    | Missing token, expired token, or invalid token                      |
| 403    | Valid token but insufficient permissions for the requested resource |

```json
{
	"error": {
		"code": "UNAUTHORIZED",
		"message": "Authentication required. Provide a valid Bearer token."
	}
}
```
